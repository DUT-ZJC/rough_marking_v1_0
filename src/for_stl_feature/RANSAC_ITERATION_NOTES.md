# for_stl_feature RANSAC 修改说明

## 1. 这次主要修了什么

当前 `method="ransac"` 走的是：

- `stl_extractor.py`
- `extractor_cgal.py`
- `cpp_moudle/cgal_ransac.cpp`

这次修复的核心目标有 3 个：

1. 解决以前提取结果呈“雪花状”的问题
2. 解决运行过程中出现的 `0xc06d007f` 原生崩溃
3. 尽量减少“平面被识别成圆柱”的误检


## 2. 以前为什么会出现雪花状

旧版本的问题本质上不是 `view.py` 画错了，而是 `ransac` 上游给出的三角片集合本来就是散的。

旧逻辑里，`CGALExtractor` 直接把：

- 三角形重心 `tri_centers`
- 三角形法向 `triangle_normals`

送进 C++ 的 CGAL RANSAC。

这样会带来两个问题：

1. CGAL 检测到的是“解析几何上属于同一个平面/圆柱”的样本点，不是“网格上连续连通”的 patch
2. 直接把 CGAL 返回的索引当成最终三角片集合时，会把大量互不相连的小三角片拼成一个 feature

所以在可视化里看起来就像“雪花状”“炸开了”。


## 3. 为什么会崩溃

你这边环境里，`cgal_ransac.pyd` 返回之后，再继续走某些 `numpy.linalg` 计算时会触发 Windows 原生层异常：

- `_fit_plane_from_points()` 里的 `np.linalg.eigh`
- `stl_extractor.py` 里圆柱二次筛选的 `np.cov + np.linalg.eigh`

所以后来我把这两类后处理改成了更保守的几何统计方式，避开了这条不稳定路径。


## 4. 这次做了哪些修改

### 4.1 C++ 层

文件：

- `cpp_moudle/cgal_ransac.cpp`

改动：

1. 给 `extract_shapes()` 增加了 `random_seed`
2. 用 `CGAL::get_default_random()` 固定随机种子

目的：

- 让同一份 STL 在同一参数下尽量可重复


### 4.2 Python 的 CGAL 提取器

文件：

- `extractor_cgal.py`

改动分 6 部分：

1. 不再直接拿三角形重心当最终 patch
2. 先对 STL 表面做采样，再把采样点送给 CGAL
3. CGAL 只负责给“平面/圆柱提议”
4. Python 侧重新把提议映射回完整 mesh 三角片
5. 映射后按网格邻接关系拆成连通域
6. 再做平面扩张、剩余区域平面回收、圆柱转平面判定

这一步的结果是：

- feature 变成网格语义上的“连续 patch”
- 不再是解析几何意义上的“散点拼接”


### 4.3 圆柱二次筛选

文件：

- `stl_extractor.py`

改动：

- 去掉了 `np.cov + eigh` 的圆柱筛选
- 改成几何规则筛选 `_is_reasonable_cylinder_feature()`

主要过滤：

1. 轴向和法向关系不对的“假圆柱”
2. 半径特别大、其实更像平面的 patch
3. 曲率分散性不足的 patch


## 5. 现在为什么“邻接平面”还可能变成两个平面

这是当前版本最重要的一个“剩余问题”。

先区分两种情况：

### 情况 A：两个平面本来就不同

比如：

- 有夹角
- 法向不完全一致
- 中间有倒角、过渡面、圆角

这种情况下，被识别成两个平面是对的。


### 情况 B：其实是同一个大平面，但被拆成两个 feature

这就是你现在看到的重点问题。

主要原因有 4 个：

1. `extractor_cgal.py` 现在会按连通域拆分
2. 某些边缘三角片先被别的 proposal 占走了，导致同一大平面被切开
3. 平面扩张是“局部扩张”，不是全局共面合并
4. 当前没有做“最终的共面 plane merge”

也就是说：

- 当前版本已经解决了“雪花状”
- 但它更偏向“宁可拆开，也不要乱并”

所以对于一整块大平面，如果中间有窄缝、薄条、局部法向波动、局部先被别的 feature 占走，就可能被分成两个 plane feature。


## 6. 为什么有些平面会被识别成圆柱

旧版本里这个问题更严重，原因是：

1. 平面没吃全
2. 剩下的边角区域又还具有一定连通性
3. CGAL 给出的某些大半径圆柱提议，会把这些平面残片吸进去

所以表现出来就是：

- 一个大平面主体识别成 plane
- 边上一部分残片却识别成 cylinder

这次我加了两层修正：

1. 平面扩张和剩余区域平面回收
2. 如果某个 cylinder component 用平面解释得更好，就直接转成 plane

所以这个误检已经明显减少了。


## 7. 当前版本的策略特点

当前版本是一个“偏保守”的版本：

- 优点：不再雪花状，稳定，不容易把大量离散三角片错误并成一个面
- 缺点：对“同一大平面的自动合并”还不够激进

换句话说，当前版本优先保证：

1. 连通性正确
2. 几何拟合合理
3. 不崩溃

但还没有完成最后一步：

- 把相邻且共面的 plane feature 合并成一个更大 plane


## 8. 共面平面合并（plane merge）

这一轮已经补上了基础版 `plane merge`，位置在：

- `extractor_cgal.py`

当前做法是：

1. 所有 plane feature 提取完成后，再做最终 merge
2. 只有满足以下条件的 plane 才允许合并：
3. 法向夹角足够小
4. 两个平面的解析距离足够接近
5. 两个 patch 在网格上相邻，或者 AABB 间隔足够小
6. 合并后重新估计平面，误差仍然在阈值内
7. 如果合并后反而变成“不连通”，则回退，不强行合并

这样做的目的，是把：

- 本来属于同一大平面的相邻 patch

重新并成一个 plane，同时尽量避免把：

- 虽然方向接近，但实际不应该合并的 patch

错误并在一起。

当前版本的 merge 是“保守 merge”：

- 优点：不会明显破坏前面已经修好的稳定性
- 缺点：还不是全局最优 merge，某些被细缝切开的共面区域，仍可能保持分开


## 9. 这次迭代结论

这次不是简单“调参数”，而是把 `ransac` 链路从“解析图元索引直接拿来用”改成了“CGAL proposal + mesh-aware rebuild”。

因此当前版本相对旧版本，已经完成：

1. 去雪花状
2. 去原生崩溃
3. 显著减少平面误识别成圆柱
4. 增强平面覆盖率
5. 提高结果稳定性

当前剩余最主要的问题不再是“完全没有共面合并”，而是：

- 现在已经有基础版 plane merge
- 但它仍然偏保守
- 某些复杂断裂情况下，同一大平面仍可能被拆成多个 feature

如果还要继续往前优化，下一步更适合做的是：

- 更强的全局 plane merge
- 或者基于边界/缝隙语义的 plane stitching
